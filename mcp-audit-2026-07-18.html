Details
Colliding tool names: 'read_file' (claude-code:evil, claude-code:benign). Calls addressed by name alone may be routed to either server.
Threat
When names collide, the client or model may route a call to the wrong server. A malicious server that shadows a trusted tool name (e.g. read_file) receives the arguments meant for the real one.
Why this matters
Name routing is the only addressing MCP tools have; collisions are ambiguity an attacker can occupy. Cross-server visibility is exactly what a config-level auditor can check and a single server cannot.
Remediation
Remove or rename one side of each collision, or drop the server you trust less. Verify which server actually serves the shadowed name in your client.

Details
Matched verb(s): delete. The tool declares neither readOnlyHint nor destructiveHint, so clients cannot single it out for confirmation.
Threat
Clients use annotations to decide when to ask the human. An unannotated destructive tool gets called with the same ceremony as a read-only one.
Why this matters
Annotations are the only machine-readable safety contract MCP tools carry. Their absence on a destructive capability removes the client's chance to interpose a confirmation.
Remediation
Prefer servers that annotate destructive tools (destructiveHint: true) and mark read-only ones (readOnlyHint: true). Treat calls to this tool as requiring manual review.
